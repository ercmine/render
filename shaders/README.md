# render shader pipeline

Statement 10 establishes the canonical, engine-owned shader workflow.

## Source tree

- `shaders/includes/` shared include files (`*.sh`) and varying definitions (`*.def.sc`)
  - `core/` common transforms/constants/macros used across shader families
  - `debug/` debug-only helpers
- `shaders/materials/<shader_name>/` material shader sources (`vs.sc`, `fs.sc`, optional `cs.sc`)
- `shaders/post/<shader_name>/` post-process shader sources
- `shaders/debug/<shader_name>/` debug visualization shader sources
- `shaders/shaders.cmake` canonical manifest of programs, variants, and defines

Generated output root (build tree):

- `bin/shaders/bin/<backend>/<category>/<program>/<variant>/<stage>.bin`
- `bin/shaders/metadata/<backend>/<category>/<program>/<variant>/`
  - `<stage>.json` stage metadata
  - `program.json` engine-facing program reflection record

Installed/staged runtime layout mirrors the same under `shaders/bin` and `shaders/metadata`.

## Naming conventions

- Stage files: `vs.sc`, `fs.sc`, `cs.sc`.
- Program identity is manifest-driven: `program`, `category`, `variant`.
- Include files use `*.sh`; varyings use `*.def.sc`.
- Stage entry point is `void main()`.
- Shared transform uniform is currently `u_modelViewProj`.

## Include rules

- Includes are resolved deterministically from:
  1. `shaders/`
  2. `shaders/includes/`
- Path style in source should be repo-relative from `shaders/` (e.g. `../includes/core/common.sh`).
- Layering rule: category shaders may include `includes/core/*`; debug shaders may also include `includes/debug/*`.

## Variants and defines

Variants are explicitly listed in `shaders/shaders.cmake` as manifest entries:

`program|category|vertex_source|fragment_source|variant|defines`

Current variant policy:

- `default` for baseline behavior
- targeted debug variants (example: `debug_tint`)
- no free-form runtime define explosion; each variant must be declared in the manifest

## Build integration

Use `render_shaders` to compile all declared programs/variants for selected backends.

Relevant CMake cache knobs:

- `RENDER_BGFX_SHADERC`: explicit `shaderc` executable path
- `RENDER_SHADER_BACKENDS`: override backend list (`dx11;spirv;glsl;metal`)
- `RENDER_SHADER_OUTPUT_ROOT`: generated output root (`<build>/bin/shaders`)
- `RENDER_REQUIRE_SHADER_COMPILATION`: fail hard when toolchain is unavailable

`render_shader_check` depends on `render_shaders` and is used by validation/CI.

## Backend/platform mapping

Compilation mapping is explicit in `cmake/compile_shaders.cmake`:

- `spirv` -> `--platform linux --profile spirv`
- `glsl` -> `--platform linux --profile 330`
- `metal` -> `--platform osx --profile metal`
- `dx11` -> `--platform windows --profile vs_5_0/ps_5_0`

Host-default backends are selected by CMake (`dx11` on Windows, `metal` on macOS, `spirv;glsl` on Linux) and can be overridden.

## Hot reload behavior

`ShaderProgramLibrary` watches timestamps of:

- vertex binary
- fragment binary
- `program.json` metadata

When any changes during runtime, it attempts to rebuild the program handle. On failure, the previous valid handle stays active.

## Reflection metadata ownership

Metadata is generated by the shader pipeline and consumed via engine paths (`shaders/metadata/...`), not by exposing raw bgfx tool output as engine API.

Current metadata captures:

- schema version
- program/category/variant/backend/stage identity
- source path
- compiled binary relative path
- define string
- SHA256 of compiled stage binary

This is the foundation for future uniform/resource binding validation.
